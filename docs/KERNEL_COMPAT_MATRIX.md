# Kernel Compatibility Matrix

> **Status:** plan + canonical manifest. Implements the deferred "multi-kernel
> matrix" item of [ENFORCEMENT_WEDGE_STRATEGY.md](ENFORCEMENT_WEDGE_STRATEGY.md).

GitHub-hosted CI runs a single kernel, so it cannot tell us whether enforcement
works on the kernels operators actually run. This document defines the target
matrix and how we verify it, and is backed by a generated manifest so the plan
cannot drift from the source of truth.

## Two layers (never conflate them)

| Layer | Question | Needs | Evidence |
|-------|----------|-------|----------|
| **A — object compat** | does `aegis.bpf.o` load, verify, and attach its hooks on kernel X? | kernel + BTF | per-kernel load/attach verdict |
| **B — behavioral** | does a denied syscall return `-EPERM` per class on kernel X? | full daemon + root + cgroups on kernel X | `enforcement_proof.sh` exit 0 |

A green Layer-A result means *"the object is compatible with kernel X"* — it is
**not** proof of enforcement. Only a Layer-B pass earns *"enforcement proven on
kernel X."* These stay distinct evidence classes in the posture contract.

## Target kernels

LTS-anchored, plus the two most-deployed Ubuntu kernels, plus one deliberately
no-BPF-LSM target:

| Version | BPF-LSM | Why |
|---------|---------|-----|
| 5.15 | yes | Ubuntu 22.04 LTS — realistic floor; LSM hook availability breaks here first |
| 6.1  | yes | Debian 12 LTS |
| 6.6  | yes | LTS |
| 6.8  | yes | Ubuntu 24.04 LTS — large install base |
| 6.12 | yes | latest LTS |
| no-lsm | **no** | a kernel without `bpf` in `lsm=` — proves the **No-Pretend invariant** (daemon refuses enforce / degrades to audit; harness exits 77) |

## The BPF-LSM boot gotcha

The most common reason BPF-LSM enforcement silently fails on a "supported"
kernel: `CONFIG_BPF_LSM=y` is necessary but **not sufficient**. The kernel must
also boot with `bpf` in the `lsm=` list (e.g.
`lsm=lockdown,capability,landlock,yama,apparmor,bpf`). Verify at runtime:

```sh
grep -q bpf /sys/kernel/security/lsm   # must succeed on a Layer-B kernel
```

Every `bpf_lsm: true` row in the matrix must satisfy this; the `no-lsm` row must
not (that is the point of it).

## Canonical manifest (generated)

`tests/enforcement/kernel_compat_manifest.json` is generated by
`scripts/gen_kernel_compat_manifest.py` from the source of truth
(`src/hook_capabilities.cpp` for programs/attach symbols, and
`tests/enforcement/enforcement_classes.tsv` for ENFORCED classes). It is
backend-neutral: it lists the artifact, the programs + `bpf_lsm_*` attach symbols
+ required flags, the enforcement classes, the kernel matrix, and the Layer-A /
Layer-B gates.

CI keeps it honest: the `enforcement-proof-contract` job runs
`gen_kernel_compat_manifest.py --check` and fails if the committed manifest is
stale relative to the source of truth.

## Backend integration (bpfcompat)

The matrix is executed by [bpfcompat](https://github.com/ErenAri/adaptive-bpf-runtime-program),
which boots each profile as a QEMU/KVM guest and runs an in-VM libbpf validator
(real load/verify/attach in guest kernel context).

**Adapter (done).** `scripts/gen_bpfcompat_manifest.py` generates
`tests/enforcement/bpfcompat-manifest.yaml` (bpfcompat's manifest schema) from the
BPF `SEC()` declarations — so it cannot drift from the object. All programs are
`required: false`; the authoritative per-kernel signal is each
`validator-result.json`'s per-program `load_status` (added to the validator).
`tests/enforcement/bpfcompat-matrix.yaml` pins the LTS profile set. CI runs
`gen_bpfcompat_manifest.py --check` (in `enforcement-proof-contract`) to keep the
committed manifest in sync. Run it:

```sh
bpfcompat test --artifact build/aegis.bpf.o \
  --manifest tests/enforcement/bpfcompat-manifest.yaml \
  --matrix   tests/enforcement/bpfcompat-matrix.yaml \
  --out report.json
```

This already produced the per-hook × per-kernel matrix that found the
`handle_execve` (≤ 6.8) and `handle_exit` (5.15) load bugs.

**Phase 1 — Layer A (object compat).** The above, gated in CI. Closes the
cross-version CO-RE / BTF / verifier / hook-availability risk single-kernel CI
cannot. **Blocked only on the runner decision** (see below).

**Phase 2 — Layer B (behavioral).** If the backend can run an arbitrary root
workload per kernel, run `enforcement_proof.sh` there and gate on exit 0 (exit 77
on the `no-lsm` profile). Until then, Layer B runs on self-hosted / VM runners via
`enforcement_proof.sh` directly.

**CI submission job (the remaining piece).** Needs an execution path + secrets:
either (a) a self-hosted runner with KVM + the cached images, invoking the CLI
above, or (b) the bpfcompat HTTP API (`bpfcompat serve`) with `BPFCOMPAT_API_*`
tokens in GitHub Actions secrets. The manifest/matrix/object are all ready; this
is a thin wrapper once the runner/API path is chosen.
